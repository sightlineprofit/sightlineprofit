# AGENTS.md

## Cursor Cloud specific instructions

**Product:** Sightline — a financial-architecture/practice-management SaaS for interior-design firm owners. Single app (not a monorepo): TanStack Start (SSR) + TanStack Router + React 19, built with Vite, styled with Tailwind v4 + Radix/shadcn. Backend is a **remote hosted Supabase** project (Postgres + Auth) plus optional Stripe billing and Lovable Cloud email/OAuth. Generated by Lovable (`@lovable.dev/*`).

**Package manager:** `bun` is the project's preferred manager (`bun.lock`, `bunfig.toml`), but `package-lock.json` is also committed and `npm` works fine for dev. The startup update script uses `npm install` because npm is always available; `bun install` also works if `bun` is on PATH.

**Standard commands** (see `package.json` scripts; run with `npm run …` or `bun run …`):
- Dev server: `npm run dev` → serves on `http://localhost:8080/` (Vite fixes the port/host).
- Build (dev mode): `npm run build:dev` (produces a Nitro/Cloudflare-Workers build under `.output/`). `npm run build` is the prod build.
- Lint: `npm run lint`. Format/autofix: `npm run format` (Prettier).
- There is **no automated test runner** configured (no `test` script, no test files).

**Supabase project:** The active project is **`nizjqvbxrmxkkmnnqzpy`** (`.env` and `supabase/config.toml` point here; `.env` was updated from the stale `ncekwpeojcutmunadrbj` it originally shipped with). `.env` holds only the public URL/project-id + the publishable (anon) key — safe to commit. New Supabase keys are used: the client publishable key is `sb_publishable_…` (in `.env`), and the server secret key is `sb_secret_…`.

**Required secret:** `SUPABASE_SERVICE_ROLE_KEY` must be set (as an env var/secret) to the `sb_secret_…` key for project `nizjqvbxrmxkkmnnqzpy`. Every TanStack server function (`src/lib/*.functions.ts`) uses the admin client in `src/integrations/supabase/client.server.ts`, which throws `Missing Supabase environment variable(s): SUPABASE_SERVICE_ROLE_KEY` when unset. Without it the `/register` pricing quote fails ("Fetching pricing…" toast) and firm creation in `/post-auth`, billing, admin, team, etc. all fail. Client-side auth (login/signup) works without it (uses the publishable key). Sanity-check a key against the project: `curl -s -o /dev/null -w "%{http_code}\n" https://nizjqvbxrmxkkmnnqzpy.supabase.co/auth/v1/settings -H "apikey: $KEY"` → non-`401`.

**Non-obvious gotchas:**
- `npm run lint` currently exits non-zero with thousands of **pre-existing** Prettier/ESLint errors across the repo — this is the repo's current state, not a setup failure. A non-zero exit here is expected; do not treat it as a regression.
- The publishable key + URL/project-id are committed in `.env`, and Vite uses those `.env` values even when an empty `VITE_SUPABASE_PUBLISHABLE_KEY` env var is present (verified: the client still resolves to `nizjqvbxrmxkkmnnqzpy` + the publishable key). A non-empty but *wrong* `VITE_SUPABASE_*` env var could still override `.env`; if the client ever points at the wrong project, `unset VITE_SUPABASE_PUBLISHABLE_KEY VITE_SUPABASE_URL VITE_SUPABASE_PROJECT_ID` before `npm run dev`.
- The Supabase project **requires email confirmation** (`mailer_autoconfirm=false`): `signUp` returns `confirmation_sent_at`, and login before confirming returns `email_not_confirmed`. For end-to-end testing create a pre-confirmed user via the admin API: `supabase.auth.admin.createUser({ email, password, email_confirm: true })` using the `sb_secret_…` key.
- **Payment gate:** a freshly created firm has no `stripe_subscription_id`, so `/post-auth` always routes new users to `/register?step=payment`, which needs Stripe (`VITE_PAYMENTS_CLIENT_TOKEN` + server Stripe secrets) and otherwise shows "Something went wrong". To reach `/onboarding` → `/dashboard` without Stripe, set the firm's `stripe_subscription_id`/`subscription_status='active'`. The `prevent_firm_billing_changes` trigger blocks writes to billing columns unless `current_setting('request.jwt.claim.role')='service_role'`; the new `sb_secret_` key does not populate that GUC via PostgREST, so update over a direct DB session with `set request.jwt.claim.role='service_role';` first.
- **DB migrations / schema:** schema is defined in `supabase/migrations/*.sql` and tracked in `supabase_migrations.schema_migrations`. If server functions fail with `Could not find the '<col>' column` the project schema is behind — apply pending migrations. The **direct** DB host (`db.<ref>.supabase.co`) is **IPv6-only and unreachable** from this VM; use the **Session/Transaction pooler** connection string (`aws-0-<region>.pooler.supabase.com`, user `postgres.<ref>`). Note: some migrations `REVOKE`/`ALTER` Lovable-provisioned functions (`email_queue_dispatch`, `email_queue_wake`) that may not exist here — those statements can be safely skipped (email-queue plumbing is optional).
- Optional integrations (not needed for core UI) require extra secrets: Stripe billing/checkout needs `STRIPE_SANDBOX_API_KEY`/`STRIPE_LIVE_API_KEY`, `PAYMENTS_*_WEBHOOK_SECRET`, `LOVABLE_API_KEY`; Lovable email/OAuth needs `LOVABLE_API_KEY` (+ service role key for the email queue worker).
- Routing is **file-based** (`src/routes/`); `src/routeTree.gen.ts` is auto-generated (regenerated on dev/build) — do not edit it by hand or commit incidental changes to it.
- Do not add `tanstackStart`, `viteReact`, `tailwindcss`, `nitro`, etc. plugins to `vite.config.ts`; they are already provided by `@lovable.dev/vite-tanstack-config` (adding them causes duplicate-plugin breakage).
